# PikPak remote-download feature — design

**Date:** 2026-07-12
**Status:** approved (architecture decisions confirmed)

## Goal

Telegram bot command `/pikpak <remote_path>` that pulls a file from the pre-configured
rclone `pikpak` remote onto the VPS and feeds it into the **existing** upload-to-Telegram
pipeline, plus `/jobs` to list recent download jobs and `/ls [folder]` to browse the remote.
Oversized files are rejected up front; no splitting anywhere.

## Key facts (verified against code, not docs)

- Upload to Telegram is done by **`watcher.py` via Telethon (MTProto user session
  `worker.session`)**, driven by the `upload_jobs` Postgres table — **not** by a filesystem
  folder watch. "Move into the monitored folder" = **insert an `upload_jobs` row**.
- `upload_jobs` lifecycle: inserted `queued` → flipped to `pending` → `watcher.claim_next()`
  grabs `status='pending'`. PikPak inserts its row **directly as `pending`** so it uploads
  automatically with no manual "Start".
- `origin='upload', cleanup_source=1` + `source_path` = a staging **directory** holding the
  one file → watcher uploads it whole and deletes the dir afterwards.
- Ceiling: MTProto user account = **2 GB** non-premium. Reject threshold `PIKPAK_MAX_BYTES`
  default 2 GB (configurable). Because oversized is rejected, every accepted file is exactly
  **one part** — no split path is ever taken.
- Migrations: `schema.sql` (initdb, fresh volume only) **and** an idempotent
  `CREATE TABLE IF NOT EXISTS` in `bot.py:post_init()` so existing DBs get the new table on
  the next boot. NOTIFY triggers follow the `upload_changed` pattern.
- rclone lives on the **host**; the container running it needs the binary + host
  `rclone.conf` bind-mounted.

## Architecture (chosen: worker in the bot process)

No new container, process, or session file. Everything lives in the `bot` container.

```
/pikpak <path>  ──► bot handler (pikpak.py)
                     1. rclone lsjson pikpak:<path>  → validate + size
                     2. size > MAX?  → reject reply, no job created
                     3. INSERT download_jobs(status='queued')
                     4. enqueue id on an asyncio.Queue; reply progress msg
                          │
   in-bot worker task ◄───┘  (concurrency = PIKPAK_MAX_CONCURRENT, default 1)
     5. status='downloading'
     6. rclone copy pikpak:<path> <staging/<jobid>>/ -P --stats=1s --stats-one-line
          parse "xx%" → update download_jobs.progress (NOTIFY pikpak_changed)
          retry <=3 with backoff on non-zero exit
     7. status='downloaded' → INSERT upload_jobs(origin='upload',
          cleanup_source=1, status='pending', source_path=<staging/<jobid>>)
     8. status='done'.  Watcher takes over → file lands in Telegram/drive.
     on failure: wipe staging/<jobid>, status='failed', error set.

  Progress-message editor: a per-job asyncio task edits the Telegram reply,
  throttled to 1 edit / >=6 s, reading download_jobs row.
```

Restart safety: on boot, `post_init` requeues any `download_jobs` left in
`downloading`/`queued` back to `queued` and the worker re-picks them (rclone `copy` is
resumable/idempotent into the same dir).

**Deliberate simplification (`ponytail:`):** the worker is one asyncio task, not a daemon.
The DB queue makes it restart-safe; if isolation is ever needed it lifts into a
`pikpak_worker.py` container using the same table + code.

## `download_jobs` table

```sql
CREATE TABLE IF NOT EXISTS download_jobs (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    source      TEXT NOT NULL DEFAULT 'pikpak',   -- room for other remotes later
    remote_path TEXT NOT NULL,
    filename    TEXT NOT NULL,
    size        BIGINT NOT NULL DEFAULT 0,
    status      TEXT NOT NULL DEFAULT 'queued'
                  CHECK (status IN ('queued','downloading','downloaded','uploading','done','failed')),
    progress    INTEGER NOT NULL DEFAULT 0,       -- 0..100
    error       TEXT,
    chat_id     BIGINT,     -- Telegram chat to edit the progress message in
    message_id  BIGINT,     -- the progress message being edited
    created_at  TEXT NOT NULL DEFAULT now_text(),
    updated_at  TEXT NOT NULL DEFAULT now_text()
);
CREATE INDEX IF NOT EXISTS idx_download_jobs_status ON download_jobs(status);
```

After handoff the worker spawns `_track_upload`, which polls the handed-off `upload_jobs` row
to its terminal state → sets `download_jobs` `uploading`→`done`/`failed` and confirms the
staging file is gone. The actual deletion is guaranteed by `cleanup_source=1` (the watcher
`rmtree`s the staging dir after a successful upload); the file is safe in Telegram at that
point and indexing reads from Telegram, not the local copy — so deleting at upload-success is
correct even before indexing completes.

## New env vars (bot_config.py + .env.example + compose)

| var | default | meaning |
|-----|---------|---------|
| `PIKPAK_REMOTE` | `pikpak` | rclone remote name |
| `PIKPAK_MAX_BYTES` | `2147483648` | reject above this (2 GB) |
| `PIKPAK_STAGING_DIR` | `/staging/_pikpak` | download staging root (on shared volume) |
| `PIKPAK_MAX_CONCURRENT` | `1` | active downloads |
| `PIKPAK_RETRIES` | `3` | rclone retries before fail |
| `RCLONE_BIN` | `rclone` | binary path |

Host rclone config dir bind-mounted into the bot container at `/root/.config/rclone:ro`
(compose var `RCLONE_CONFIG_DIR`, default `/home/ec2-user/.config/rclone`).

## Error handling

- rclone binary missing / remote missing / auth fail (lsjson exits non-zero or unknown
  remote) → reply: run `rclone config` on the VPS. No job created.
- Path not found (empty lsjson) → "path not found on PikPak".
- Points at a directory (lsjson returns >1 entry / IsDir) → reject with a clear message
  (single-file only in v1).
- Oversized → `File is X GB, exceeds the 2 GB Telegram limit` — no download.
- rclone copy non-zero after retries → `failed`, staging wiped, error surfaced in `/jobs`
  and the progress message.

## `/ls [folder]` — browse the remote

`rclone lsf pikpak:<folder>` (root if omitted). Directories get a trailing `/` (rclone
default). Reply is a code block; truncate to ~50 entries with a `… (N more)` footer. Same
failure handling as `/pikpak` (missing remote/auth → run `rclone config`). Read-only, no job.

## Access control

`/pikpak`, `/jobs`, and `/ls` gated to owner + authorized users (same guard the other
command handlers use).

## One runnable check

`bot/test_pikpak.py` (assert-based, no framework): rclone `--stats-one-line` percent parser
+ the size-reject decision. Runs standalone.

## Out of scope (v1)

Directory/recursive downloads, cancel command, multiple remotes UI, splitting. No streamer
or dashboard changes beyond the shared DB migration.
